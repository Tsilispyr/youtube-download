# YT → MP3 — Developer Documentation

---

## What this app does

A self-hosted web app that converts YouTube and YouTube Music URLs to MP3 files and delivers them directly to the user's device. No account required, no files stored on the server.

**Single video** → one `.mp3` Save dialog in the browser  
**Playlist** → all tracks fetched and unpacked individually into the browser (no zip on device)

## Architecture overview

```
Browser
  │
  ├─ POST /api/info      → yt-dlp extracts metadata (no download)
  ├─ POST /api/download  → starts background thread, returns session_id
  ├─ GET  /api/progress  → Server-Sent Events stream (real-time progress)
  ├─ GET  /api/serve     → streams single MP3 to browser, deletes from server
  └─ GET  /api/serve-zip → streams playlist zip to browser, deletes from server

Server (Flask)
  │
  └─ yt-dlp
       │
       └─ bgutil PO token server (localhost:4416)
              │
              └─ Headless Chromium generates YouTube auth tokens
```

## Key technical decisions

### Why yt-dlp?

The de-facto standard for YouTube downloading. Actively maintained, handles all URL formats (youtube.com, music.youtube.com, playlists, mixed URLs), manages audio extraction, metadata embedding, and thumbnail embedding in one pipeline. Alternative libraries are either abandoned or far less capable.

### Why Proof-of-Origin tokens instead of cookies?

YouTube now requires either a logged-in session cookie or a PO token to serve video streams. Cookies tie downloads to a real user account, which risks bans and requires manual refresh every few weeks. PO tokens are generated by a headless browser session without any account — they prove the request comes from a real browser environment without identifying any user. This is the approach the yt-dlp community converged on in 2024.

### Why bgutil-ytdlp-pot-provider?

It's the most maintained PO token plugin for yt-dlp. It exposes a local HTTP server on port 4416 that yt-dlp calls automatically via the plugin. The plugin is installed as a Python package — yt-dlp discovers it at runtime without any code changes needed.

### Why clone and build bgutil instead of using npm install-server?

The `bgutil-ytdlp-pot-provider install-server` npm convenience command downloads a pre-built Chromium binary at runtime. This is unreliable in Docker builds — it times out, fails silently, or requires system libraries not present in slim images. Cloning the TypeScript source and compiling it ourselves with `npm ci && npx tsc` is reproducible, offline-safe after clone, and produces a plain `build/main.js` that runs with just `node`.

### Why Server-Sent Events for progress?

Downloads take time and the frontend needs live per-track status. SSE is a one-way push from server to browser over a single HTTP connection — simpler than WebSockets (no handshake, no library, native browser support everywhere), and better than polling (no repeated requests, no latency). Each download session gets its own `queue.Queue` on the server fed by the download thread.

### Why per-session temp directories?

Each download batch gets an isolated folder under `/tmp/yt-mp3-sessions/<session_id>/`. This means multiple concurrent users don't interfere with each other, and cleanup is trivial — delete the folder after serving. Nothing accumulates on the server between sessions.

### Why zip for playlists server-side?

yt-dlp downloads tracks sequentially. Sending them to the browser one by one as they finish would mean the browser opens Save dialogs mid-download, which is confusing. Zipping on the server means one efficient transfer over the wire, then the browser unpacks it. The zip is a transport mechanism only — it never lands on the user's device.

### Why unzip in the browser with JSZip instead of serving the zip directly?

Browsers cannot create folders in the local filesystem. If we served the zip directly, the user would have to manually unzip it. JSZip fetches the zip blob, unpacks it entirely in memory, and triggers individual Save dialogs per file (MP3s first, cover last), staggered 400ms apart so the browser doesn't block rapid-fire downloads. The result: files appear in the user's Downloads folder with no manual step.

### Why not use the File System Access API to write a real folder?

The File System Access API lets JavaScript write into a user-chosen directory, which would be ideal. However it's only supported on Chrome and Edge (desktop), not Safari, Firefox, or any mobile browser. JSZip + individual saves works on all browsers and all platforms including iPhone and Android.

### Why a single HTML template file?

All CSS and JS live in one file in `templates/`. The Flask `index()` route scans the templates folder and serves whatever `.html` file it finds — no hardcoded filename. This means swapping the entire UI is a one-file drop-in with no code changes. It also makes the project easy to fork with a custom frontend.

### Why Flask over FastAPI/Django?

This app has no database, no auth, no ORM, no admin. Flask is minimal, ships SSE with just `Response(generate(), mimetype="text/event-stream")`, and handles background threads naturally. FastAPI would add async complexity without benefit here.

## Bot detection bypass — how it works end to end

```
1. yt-dlp makes a request to YouTube
2. bgutil-ytdlp-pot-provider Python plugin intercepts it
3. Plugin calls localhost:4416 (the bgutil Node.js server)
4. bgutil server runs a headless Chromium session
5. Chromium generates a valid PO token (proof the request is browser-like)
6. Token is returned to yt-dlp and attached to the YouTube request
7. YouTube accepts the request
```

Tokens expire and are refreshed automatically by the server. No manual intervention needed.

## Deployment

### Local — Docker (recommended for testing)

```bash
docker compose up --build
```

Opens on `http://localhost:5000`. The Dockerfile builds everything from scratch including the bgutil server. Subsequent starts reuse the image — only `--build` rebuilds.

To upgrade yt-dlp without rebuilding everything:
```bash
docker compose exec yt-mp3 pip install -U yt-dlp
```

### Local — without Docker

Requirements: Python 3.10+, Node.js 18+, ffmpeg, git.

```bash
# Install ffmpeg
# macOS:   brew install ffmpeg
# Ubuntu:  sudo apt install ffmpeg
# Windows: https://ffmpeg.org/download.html

pip install -r requirements.txt

# start.sh clones and builds bgutil automatically on first run, then starts everything
./start.sh
```

`start.sh` is self-healing — it checks for each dependency and installs anything missing before starting. Safe to re-run.

### Render.com (from GitHub)

1. Push repo to GitHub
2. Render.com → **New → Web Service** → connect repo
3. Render detects `render.yaml` and configures the service automatically
4. Every push to the default branch triggers an auto-redeploy

`render.yaml` uses Docker runtime so the same Dockerfile runs on Render as locally. The `PORT` env variable is injected by Render at runtime — `app.py` reads it with `int(os.environ.get("PORT", 5000))`.

**Important:** Render's free tier spins down after 15 minutes of inactivity. The bgutil token server starts fresh on each spin-up via `start.sh`. Paid plans keep the container alive.

## How start.sh works

```
1. Locate bgutil build (Docker path → home dir → local dir)
2. If not found: git clone + npm ci + tsc (one-time, ~2 min)
3. Check Python packages — pip install if missing
4. Check ffmpeg — warn if missing
5. node build/main.js &   (token server in background)
6. nc -z localhost 4416   (poll until port accepts connections, max 30s)
7. exec python app.py     (replace shell as PID 1 for clean signal handling)
```

`exec` is important — it replaces the shell process so Flask becomes PID 1 and receives Docker stop/kill signals correctly.

## How to upgrade bgutil

In the Dockerfile, change the `BGUTIL_VERSION` ARG:

```dockerfile
ARG BGUTIL_VERSION=1.3.2   # ← bump this
```

Then rebuild:
```bash
docker compose build --no-cache && docker compose up
```

Or on Render, update the ARG and push to GitHub.

---

## Updating yt-dlp

YouTube changes its extraction logic frequently. When downloads start failing:

```bash
# Docker / Render
docker compose build --no-cache && docker compose up

# Local
pip install -U yt-dlp
```

On Render, simply push any commit — the build installs the latest `yt-dlp>=2024.1.1` from requirements.txt each time.

## The docker config.json warning

```
WARNING: Error parsing config file (/home/user/.docker/config.json):
invalid character 'ï' looking for beginning of value
```

This is a local Docker credential store encoding issue on the host machine — nothing to do with the app. It's a warning only and does not affect the build or runtime. To fix it: open `~/.docker/config.json` and ensure it's valid UTF-8 JSON (usually caused by a corrupted or empty file). Replace it with `{}` if needed.

## Notes

- MP3s are 320 kbps with embedded metadata (title, artist) and album art thumbnail
- Playlist cover image is fetched from the playlist's own thumbnail, falling back to the first track's thumbnail
- Temp files live in `/tmp/yt-mp3-sessions/<session_id>/` and are deleted immediately after the browser download starts
- The `templates/` folder is scanned alphabetically — if you add multiple HTML files, the first alphabetically wins
- `requirements-test.txt` contains pytest dependencies for unit testing
